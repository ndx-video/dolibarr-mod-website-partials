# Local module-dev Dolibarr stack

## Summary

Stopped the running `dolibarr-braypark` stack on gandalf and added a lean Dolibarr 22 + Postgres module-dev environment under this repo’s `dev/`, served at `https://partials.gandalf.lan` via Caddy on `gandalf_proxy`. The module tree is bind-mounted for live development; REST API and Website modules are enabled on first install.

## Changes

- Stopped `dolibarr-braypark` via `scripts/stop.sh` (volumes preserved; `dlb` / `sndlb` Caddy blocks left in place)
- Added `dev/docker-compose.yml`, `dev/.env.example`, `dev/install.forced.php`, `dev/scripts/{up,down,install}.sh`
- Added `htdocs/custom/websitepartials/.gitkeep` as the bind-mount target
- Added Caddy site blocks for `partials.gandalf.lan` in `/opt/caddy/Caddyfile` plus `dev/caddy/partials.gandalf.lan.caddy`
- Added `.gitignore` (`dev/.env`); documented local stack in `README.md` and `AGENTS.md`

## Decisions

- Keep module-dev separate from Sentinel (`dolibarr-braypark` / `dlb.gandalf.lan`); use `partials.gandalf.lan` so both can coexist later.
- No host `ports:` — ingress only through Caddy on `gandalf_proxy` (dock convention).
- Bind-mount only `htdocs/custom/websitepartials` (not a full `custom/` volume) so the official image’s other custom paths stay untouched.
- Secrets and DB data stay local (`dev/.env` gitignored; named Docker volumes).

## Follow-ups

- Scaffold module PHP under `htdocs/custom/websitepartials/` (P0) and enable it in Setup → Modules.
- Create a Website (`main-website`) and a published page for P1 smoke tests.
- Generate a user `DOLAPIKEY` when exercising the REST control plane (P2).
