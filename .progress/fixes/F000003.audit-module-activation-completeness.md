# Fix — Audit all enabled modules for SQL-only activation gaps

## Summary

Audited every `MAIN_MODULE_*=1` const on the module-dev stack for incomplete activation (declared menus/rights vs rows in `llx_menu` / `llx_rights_def`, and `_load_tables` usage). **Only Website was incomplete** (tables missing from the earlier SQL-only enable). Other enabled modules already had matching menus and rights.

Navbar confusion had two separate causes:

1. Empty company profile (`MAIN_INFO_SOCIETE_NOM` / `COUNTRY`) → login redirected to `setupnotcomplete`
2. Most enabled modules do **not** register top-nav entries — only **Agenda** and **WebSites** declare `type=top` menus; API / ModuleBuilder / Import / Export live under Tools left menus

## Changes

- Re-ran `activateModule` for all enabled modules (heal pass); fixed `www-data` ownership on document dirs
- Set minimal company name + AU country so the UI is not gated
- Added `dev/scripts/setup-company.php` and `dev/scripts/audit-modules.php`
- Updated `dev/scripts/up.sh` to run company setup + chown after activates; documented no SQL-only enables
- Extended `dev/.env.example` with `DOLI_COMPANY_NAME` / `DOLI_COMPANY_COUNTRYCODE`

## Decisions

- Keep forcing only `modApi`, `modWebsite`, `modWebsitePartials` in `up.sh`; other modules come from CLI install / image defaults and already look complete
- Never enable modules by flipping `llx_const` alone
